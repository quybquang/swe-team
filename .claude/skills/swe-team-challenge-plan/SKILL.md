# Challenge Plan

## Overview

A plan written immediately after a spec is optimistic by construction. The planner just
convinced itself the plan is correct — it is the worst possible reviewer of its own output.
This skill adopts the opposite frame: **every task is guilty of over-engineering until
proven innocent**. It asks the question a staff engineer asks in planning review: "Do we
actually need to do this? Is there a simpler way? Is this the right layer?"

This is NOT a correctness check (that is VERIFY's job). This is a **scope and complexity
audit** — catching waste before it becomes code.

## When to Use

- Always, immediately after `swe-team:decompose-tasks` produces `tasks.json`.
- Invoked by swe-lead as the step between PLAN and BUILD.
- Also useful after a re-plan — re-run on new tasks appended to tasks.json.

## When NOT to Use

- Do NOT re-run challenge on tasks already marked `done` or `verified` — they are committed.
- Do NOT block BUILD if all findings are advisory. Challenge output is a cost-benefit signal,
  not a hard gate (unless a `critical` finding is present — see Process Step 4).

---

## Process

### Step 1 — Load context

Read:
1. `.claude/swe-team/runs/current/tasks.json` — the plan to challenge.
2. `.claude/swe-team/runs/current/run.json` — original requirement (ground truth).
3. `.claude/swe-team/runs/current/spec.md` (if exists) — the agreed acceptance criteria.
4. `.claude/swe-team/runs/current/adversarial-notes.md` (if exists) — upstream spec findings.

For each task in tasks.json with status `pending` (not `done` / `verified`), run the four
challenge axes below.

### Step 2 — Four challenge axes (run ALL for every pending task)

**Axis 1 — Necessity**
> Is this task directly required by the original requirement, or is it inferred overhead?
> "Can we ship the requirement WITHOUT this task, even if it's less clean?"

Record: `[UNNECESSARY] T<id>: <reason this task may not be needed>`

**Axis 2 — Over-engineering**
> Does this task introduce abstraction, configuration, or generality beyond what the
> requirement demands? "Is this solving a problem we don't have yet?"

Signals: new base classes, interface extractions, factory patterns, config-driven behaviour
for a single use-case, "extensibility" not mentioned in requirement.

Record: `[OVERENG] T<id>: <specific abstraction or pattern> — not required by <req_ref>`

**Axis 3 — Wrong layer**
> Could this be solved with config, existing library, or framework feature instead of new code?
> "Are we writing code that already exists?"

Record: `[WRONG_LAYER] T<id>: could use <alternative> instead of writing <described_code>`

**Axis 4 — Task mergeability**
> Could this task be merged with an adjacent task without violating `max_files_per_task`?
> Unnecessary task splits increase context switching and inter-task dependency risk.

Record: `[MERGE] T<id> + T<id2>: both touch <shared_file_or_concern> — consider merging`

### Step 3 — Classify findings

| Severity | Meaning | swe-lead action |
|---|---|---|
| `critical` | The task as specified will introduce a wrong abstraction or significant scope creep that conflicts with the requirement | MUST replan before BUILD |
| `high` | Task can be simplified materially (>30% estimated effort savings) | SHOULD replan or constrain |
| `medium` | Improvement opportunity — task as written will work, just not optimally | Record; proceed |
| `advisory` | Stylistic — no impact on build success | Record; proceed |

### Step 4 — Write challenge-notes.md and return

Write `.claude/swe-team/runs/current/challenge-notes.md`:

```markdown
# Plan Challenge Notes

> Generated by swe-team:challenge-plan.
> swe-lead: address CRITICAL findings before BUILD. HIGH findings are strongly recommended.

## Summary
- Critical: <N>  (replan required if > 0)
- High: <N>
- Medium: <N>
- Advisory: <N>

## Per-task findings

### T<id> — <task title>
<!-- [AXIS] finding — severity justification -->

## Acknowledged overrides
<!-- swe-lead fills when proceeding despite a critical or high finding -->
```

Append event to `events.jsonl`:
```json
{"kind":"challenge_plan_complete","ts":"<iso8601>","run_id":"<id>",
 "tasks_challenged":<N>,"findings":{"critical":<N>,"high":<N>,"medium":<N>,"advisory":<N>},
 "path":"challenge-notes.md"}
```

swe-lead decision rules:
- `critical > 0` → MUST replan (`swe-team:replan`) OR write explicit acknowledgement in
  `## Acknowledged overrides` with justification. Proceeding silently is not allowed.
- `high > 0` → SHOULD replan for affected tasks OR acknowledge.
- `medium` / `advisory` → proceed; annotate PR body for reviewer.

## Anti-Rationalizations

| Excuse | Rebuttal |
|---|---|
| "Every task was derived directly from the spec." | The spec itself may have been over-specified. Check the original requirement, not the spec. |
| "The abstractions are standard engineering practice." | Standard practice applied where not needed is over-engineering. It still costs build time and maintenance. |
| "I'll skip Axis 3 — we're not using external libraries for this." | Axis 3 is about config and framework features too, not just npm packages. |
| "Merging tasks would make each task too large." | If tasks can only survive by staying separate because they'd exceed file limits when merged, that is a smell, not a reason to skip the check. |
| "There are no critical findings — the plan is fine." | No critical findings means you checked thoroughly. It does not mean the plan is optimal. |
| "I am the one who wrote the plan, so I know it's necessary." | That's exactly why this skill exists. |

## Red Flags

- Zero findings for a plan with > 5 tasks → not adversarial enough. Re-run Axis 1 and 2.
- Every finding is `advisory` for a plan that touches > 8 files → severity inflation. Review criteria.
- `## Acknowledged overrides` is empty after a `critical` finding → swe-lead skipped its obligation. Emit a `blocker` event.
- Challenge notes were written but swe-lead proceeded directly to BUILD without reading them → the skill was bypassed. Emit `stuck` event.

## Verification

After this skill completes:
- `challenge-notes.md` exists in the run dir.
- `challenge_plan_complete` event appended to `events.jsonl`.
- Finding counts in event match document.
- swe-lead has addressed all `critical` findings (replan or acknowledged override) before BUILD.
- Read-only on `tasks.json` — this skill does NOT modify tasks.
