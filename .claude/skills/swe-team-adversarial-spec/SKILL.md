# Adversarial Spec Review

## Overview

A spec written by the same AI that will implement it has a structural bias toward optimism.
This skill breaks the circular loop by explicitly adopting the opposite frame:
**the spec is wrong until proven right**. Every claim in the spec must survive challenge.

The goal is NOT to block the run — it is to surface what the spec silently assumed so that
swe-lead can either (a) refine the spec before PLAN, or (b) acknowledge the assumption
explicitly so swe-coder doesn't hit it mid-build.

## When to Use

- Always, after DEFINE produces `spec.md` (or when spec is inferred from `pre-flight-brief.md`
  without a DEFINE phase).
- Invoked by swe-lead as the mandatory step between DEFINE and PLAN.

## When NOT to Use

- Do NOT run if `phases.skip_define: true` AND no `spec.md` exists — skip with a warning
  to swe-lead that adversarial review was bypassed.
- Do NOT block the run based on adversarial notes alone. Output is advisory. swe-lead decides
  whether to replan, clarify, or proceed with documented assumptions.

## Process

### Step 1 — Load context

Read:
1. `.claude/swe-team/runs/current/spec.md` — the primary target.
2. `.claude/swe-team/runs/current/pre-flight-brief.md` — the upstream assumptions already documented.
3. `.claude/swe-team/runs/current/run.json` — original requirement as the ground truth.

Cross-read: for each statement in spec.md, trace it back to run.json. If a spec claim cannot
be traced to the original requirement, it is a candidate assumption.

### Step 2 — Challenge across five axes

Run every axis. Do NOT skip any, even if the spec looks clean.

**Axis 1 — Silent assumptions**
Find statements in spec.md that assume facts not present in run.json. Format each as:
> `[ASSUME] "<spec claim>" — not stated in requirement. If wrong, impact: <consequence>.`

**Axis 2 — Missing edge cases**
For each acceptance bullet, ask: what happens at the boundary?
- Empty input, null, zero, max value
- Concurrent access
- Unauthenticated / unauthorised actor
- Dependency unavailable (DB down, external API timeout)

If the spec is silent on an edge case that could cause a production incident, record it:
> `[EDGE] Acceptance bullet <A_N>: silent on <edge case>. Potential: <failure mode>.`

**Axis 3 — Ambiguous acceptance bullets**
An acceptance bullet is ambiguous if two different implementations could both satisfy it.
> `[AMBIG] "<bullet text>" — could mean X or Y. Implementation will silently pick one.`

**Axis 4 — Scope creep indicators**
Check if spec.md adds deliverables not in run.json (without pre-flight justification).
> `[SCOPE] spec.md introduces "<item>" — not requested. Adds risk without explicit mandate.`

**Axis 5 — Dependency on external state**
Does any acceptance bullet assume an external system, schema, or API contract that may not
match production reality?
> `[DEP] "<bullet>" assumes <schema/API/config> at <path or service>. Not validated.`

### Step 3 — Classify findings

Each finding gets a severity:

| Severity | Meaning | swe-lead action |
|---|---|---|
| `critical` | If wrong, the build will fail or produce broken behaviour | MUST clarify before PLAN |
| `high` | If wrong, acceptance criteria can pass while intent is missed | SHOULD add to spec before PLAN |
| `medium` | Edge case gap — unlikely but production-relevant | SHOULD add to spec or acknowledge |
| `low` | Style / clarity issue — implementation will probably get this right anyway | Acknowledge; proceed |

### Step 4 — Write adversarial-notes.md

Write `.claude/swe-team/runs/current/adversarial-notes.md`:

```markdown
# Adversarial Spec Review

> Generated by swe-team:adversarial-spec — hostile review, not endorsement.
> swe-lead: address CRITICAL and HIGH findings before invoking decompose-tasks.

## Summary
- Critical: <N>
- High: <N>
- Medium: <N>
- Low: <N>

## Findings

### CRITICAL
<!-- one entry per finding, format: [AXIS] description — severity justification -->

### HIGH
<!-- ... -->

### MEDIUM
<!-- ... -->

### LOW
<!-- ... -->

## Acknowledged assumptions
<!-- swe-lead fills this section when it decides to proceed despite a finding -->
```

### Step 5 — Emit event

Append to `.claude/swe-team/runs/current/events.jsonl`:
```json
{"kind":"adversarial_spec_complete","ts":"<iso8601>","run_id":"<id>",
 "findings":{"critical":<N>,"high":<N>,"medium":<N>,"low":<N>},
 "path":"adversarial-notes.md"}
```

### Step 6 — Return to swe-lead

Return a one-line summary: `"adversarial-spec: C=<N> H=<N> M=<N> L=<N>"`.

swe-lead decision rules:
- `critical > 0` → swe-lead MUST either update spec.md to address the finding OR write an
  explicit acknowledgement in the `## Acknowledged assumptions` section of adversarial-notes.md
  before calling `swe-team:decompose-tasks`. If swe-lead cannot resolve the critical finding
  with available context, trigger interactive clarify with the user.
- `high > 0` → swe-lead SHOULD address or acknowledge before PLAN. May proceed if the high
  finding can be encoded as a guard in an acceptance bullet.
- `medium` / `low` → proceed; record in PR body for reviewer attention.

## Anti-Rationalizations

| Excuse | Rebuttal |
|---|---|
| "The spec is obviously correct for this simple requirement." | Simple requirements have simple spec failures. Run all five axes regardless. |
| "I'll skip Axis 3 — all bullets look unambiguous to me." | Ambiguity is invisible to the author. Run Axis 3 every time. |
| "No critical findings — the spec is good." | Absence of critical findings means you checked. It does not mean the spec is correct. |
| "The coder will figure out the edge cases during build." | The coder will pick the path of least resistance. Edge cases not in acceptance bullets will not be verified. |
| "This finding is theoretical — it probably won't happen." | Every production incident was theoretical before it happened. Record it. |
| "I should resolve the critical finding myself." | swe-lead decides. Your role is to surface, not to fix. Write it in adversarial-notes.md and return. |

## Red Flags

- You produced zero findings for a spec longer than 10 acceptance bullets → you were not adversarial enough. Re-run Axis 1 and 2.
- Every finding is rated `low` → review your severity criteria. At least one medium per real-world feature is the norm.
- You are tempted to soften a critical finding to high to avoid blocking swe-lead → write it as critical. swe-lead has the authority to acknowledge and proceed.
- adversarial-notes.md `## Acknowledged assumptions` section is empty after a critical finding → swe-lead skipped its obligation. Flag as a `blocker` event and surface to user.

## Verification

After this skill completes:
- `adversarial-notes.md` exists in the run dir.
- `adversarial_spec_complete` event appended to `events.jsonl`.
- Finding counts in the event match counts in the document.
- No spec edits made by this skill — read-only on spec.md.
- swe-lead has either updated spec.md or populated `## Acknowledged assumptions` for every critical finding before invoking decompose-tasks.
