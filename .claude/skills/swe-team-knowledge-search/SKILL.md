# Knowledge Search

## Overview

`swe-team:knowledge-search` fans out a keyword search across every source listed in
`swe-team.config.json#knowledge.sources`, merges the results, and writes a
`knowledge-context.md` file to the run directory. The CLARIFY skill and DEFINE skill
then read this file to ground their outputs in existing business and domain context.

Each source type uses its own adapter:

| Source type | Read adapter |
|---|---|
| `local` | `grep -r` over `wiki/projects/<project_namespace>/` and `wiki/engineering/` |
| `notion` | `notion-search` MCP tool |
| `confluence` | Atlassian MCP search |
| `linear` | `list_issues` MCP filtered by keyword |

If `knowledge.sources` is empty or the file doesn't exist, the skill is a no-op — it exits
without writing `knowledge-context.md` and without emitting an error.

## When to Use

- A new run has been initialized (`run.json` exists, phase is pre-CLARIFY).
- `swe-team.config.json#knowledge.sources` contains at least one source with `"read"` in
  `capabilities`.
- `knowledge-context.md` does NOT already exist in the run dir (idempotent guard).

## When NOT to Use

- `knowledge.sources` is empty or missing from config.
- No source has `"read"` in its `capabilities`.
- `knowledge-context.md` already exists in the run dir (skip re-fetch; idempotent).
- You are NOT at the pre-CLARIFY step (don't invoke mid-BUILD or in RETRO).

## Process

### 1. Preflight

Read config and extract keyword set.

```bash
cat swe-team.config.json | jq -r '.knowledge.sources // []'
cat .claude/swe-team/runs/current/run.json | jq -r '.requirement.text // .requirement.fetched_content'
```

If `sources` is empty array, write a one-line `knowledge-context.md`:

```markdown
<!-- knowledge-search: no sources configured — skipped -->
```

Then stop (no error, no event emitted).

### 2. Extract keywords

From the raw requirement text, extract:
- Proper nouns and named entities (component names, route names, table names)
- Domain verbs (the action being requested)
- 3–5 most specific terms (avoid noise words: "the", "a", "user", "feature")

### 3. Fan-out search — one adapter per source

For each source in `knowledge.sources` where `capabilities` includes `"read"`:

#### `local` adapter

```bash
VAULT_PATH=$(cat swe-team.config.json | jq -r '.knowledge.sources[] | select(.type=="local") | .path')
NS=$(cat swe-team.config.json | jq -r '.knowledge.sources[] | select(.type=="local") | .project_namespace // ""')

# Search project namespace first (higher relevance), then engineering patterns
grep -r --include="*.md" -l "<keyword1>\|<keyword2>" \
  "$VAULT_PATH/wiki/projects/$NS/" \
  "$VAULT_PATH/wiki/engineering/" 2>/dev/null | head -8
```

For each matched file, read the first 60 lines (Summary + Key Concepts sections).

#### `notion` adapter

Invoke `notion-search` MCP with the extracted keyword string.
Request `page_size: 5`, `max_highlight_length: 300`.
If the MCP tool is unavailable, log a warning and skip this source:

```bash
jq -nc '{kind:"knowledge_search_warn",source:"notion",reason:"MCP unavailable"}' \
  >> .claude/swe-team/runs/current/events.jsonl
```

#### `confluence` adapter

Invoke the Atlassian MCP search with `space_key` from config and extracted keywords.
If the MCP tool is unavailable, log a warning and skip.

#### `linear` adapter

Invoke `list_issues` MCP with `query: "<keywords>"`, `limit: 5`.
Linear is read-only; use it to surface related tickets and their accepted outcomes.
If the MCP tool is unavailable, log a warning and skip.

### 4. Merge and rank

Collect all results. For each result compute a relevance score:
- +2 if result is from `wiki/projects/<project_namespace>/` (project-specific)
- +1 if result's title contains any extracted keyword
- +1 if result was updated within the last 90 days

Keep the top 6 results across all sources.

### 5. Write knowledge-context.md

```markdown
# Knowledge Context

> Auto-generated by swe-team:knowledge-search. Do not edit manually.
> Run: <run_id>  Keywords: <comma-separated keywords>

## Relevant Pages

<For each result, one section:>

### [<title>](<source-path-or-url>) — <source-id>

<Extracted summary or highlight, ≤ 200 words>

---
```

If no results found across any source, write:

```markdown
# Knowledge Context

> No relevant pages found for keywords: <keywords>
```

### 6. Emit event

```bash
RUN_ID=$(jq -r .run_id .claude/swe-team/runs/current/run.json)
jq -nc --arg rid "$RUN_ID" --arg ts "$(date -u +%FT%TZ)" \
  --argjson n <result_count> --argjson s <source_count_searched> \
  '{kind:"knowledge_search",ts:$ts,run_id:$rid,agent:"swe-lead",
    results_found:$n,sources_searched:$s}' \
  >> .claude/swe-team/runs/current/events.jsonl
```

## Anti-Rationalizations

| Excuse | Rebuttal |
|---|---|
| "The requirement is clear enough — I don't need knowledge context." | Run the search anyway if sources are configured. Agents don't know what they don't know. A 30-second grep can surface a domain rule that prevents a full replan. |
| "I'll search 20 keywords to be thorough." | Cap at 5 keywords. More keywords = noisier results. Prefer specific nouns over generic verbs. |
| "The Notion result looks irrelevant so I'll skip adding it." | Include it and let CLARIFY decide. The search skill does not judge business relevance — it retrieves. |
| "The local vault has nothing — no point searching Notion too." | Every configured source is searched independently. One empty source does not short-circuit others. |
| "I'll include the full page content to give more context." | Cap at 200 words per result. Long inclusions waste tokens and dilute signal. Provide the path/URL so the agent can fetch the full page if needed. |

## Red Flags

- `grep` returns >20 files for a single keyword — the keyword is too generic; narrow it.
- A Notion or Confluence result is >1 year old — flag it in the excerpt: `[Note: page not updated since <date>]`.
- `knowledge-context.md` grows beyond 2000 words — truncate to top 4 results; cap is signal that keywords are too broad.
- MCP tools unavailable for all non-local sources and local vault is also empty — write the "no results" stub and continue. Never block CLARIFY waiting for knowledge.

## Verification

After this skill completes:
- `.claude/swe-team/runs/current/knowledge-context.md` exists (even if it contains only the "no sources" or "no results" stub).
- `events.jsonl` contains a `{kind:"knowledge_search"}` event with `results_found` and `sources_searched` fields.
- The file contains at most 6 result sections.
- No full page dumps — each result section is ≤ 200 words.
- If all sources returned 0 results, the file still exists (stub form) and the event is still emitted.
