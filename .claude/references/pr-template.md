# PR template

This is the template `swe-team:open-pr` (§6) uses when drafting the PR body. Fields in `<angle brackets>` are filled in from run state.

---

```markdown
## Summary

<one-paragraph summary of what this PR changes and why, grounded in run.json.requirement>

## Changes

<bulleted list of tasks — title + one-line description>

- `T1` — <title>. <one-line>
- `T2` — <title>. <one-line>
...

## Plan

Original requirement:

> <requirement.text or fetched_content, truncated to 500 chars>

Full plan: [`.claude/swe-team/runs/<run_id>/tasks.json`](.claude/swe-team/runs/<run_id>/tasks.json)

## Verification

| Task | Mech | Sem | Commit |
|---|---|---|---|
| T1 | ✅ | ✅ | `<sha>` |
| T2 | ✅ | ✅ | `<sha>` |
...

- **Tests**: `<pass>/<total>` passing (`<delta>` new)
- **Lint**: ✅
- **Typecheck**: ✅
- **Requirement coverage**: `<pct>%`
- **Cross-task conflicts**: none

Full verification log: [`.claude/swe-team/runs/<run_id>/verification.jsonl`](.claude/swe-team/runs/<run_id>/verification.jsonl)

## Breaking changes

<either "None detected." or bulleted list derived from diff analysis>

## Commits

<`git log <base>..HEAD --oneline`>

## Run metadata

- Run ID: `<run_id>`
- Base branch: `<base>`
- Budget used: `<tokens>` tokens / `$<usd>`
- Re-plans: `<count>` of <max>

---
🤖 Generated by [swe-team](https://github.com/.../swe-team) v<version>.
```

## Rules for `swe-pr`

- Never claim verification status that isn't in `verification.jsonl`.
- Never summarize a change beyond what the diff shows.
- If `requirement_coverage_pct < 100`, state it explicitly — do not round up.
- Breaking-change detection is heuristic (signature changes in exported symbols, schema migrations) — when unsure, say "review recommended" rather than "none".
